# NarraLang 功能性发展方向评估

## 目的

这份文档补充 `project/docs/2026-04-06-architecture-evaluation-and-roadmap.md` 的“概述性结论”，转而回答更具体的问题：

- 以当前仓库为基础，NarraLang 接下来最应该补哪些功能。
- 哪些方向属于“缺失但必要”的能力。
- 哪些方向能形成更强的产品竞争力。
- 哪些仓库内工具值得做，且不依赖外部仓库。

本文只讨论 **本仓库内可落地** 的能力与工具，不包含 VSCode 扩展等仓库外项目。

## 评估方法

本结论基于以下事实来源交叉得出：

- `README.md`
- `docs/zh/0. 基本.md`
- `docs/zh/index/数据交互特性总结.md`
- `docs/zh/2. 数据交互/11. 控制流.md`
- `docs/zh/2. 数据交互/14. 内建语法糖.md`
- `project/docs/2026-04-06-architecture-evaluation-and-roadmap.md`
- `src/core/**`
- `src/runtime/**`
- `test/parser/**`
- `test/lexer/**`

评估逻辑很简单：

1. 先看文档承诺了什么。
2. 再看 parser 实际支持了什么。
3. 然后看 runtime 和未来 compiler 真正能兑现多少。
4. 最后筛掉那些“现在看起来很酷，但会妨碍闭环”的方向。

## 当前功能缺口图

在进入未来方向之前，先把当前最明显的功能缺口说明白。

### 1. 文档能力远大于 runtime 落地能力

目前 `docs/zh` 已经定义了较多语言能力：

- 变量与作用域
- 数组、对象、元组、字符串
- 函数、宏函数、`await`
- 控制流与迭代
- 内建对象
- 内建语法糖
- 富文本对白

但 runtime 当前的兑现度很低：

- `src/runtime/index.runtime.ts` 为空。
- `src/runtime/bridge/Bridge.ts` 为空。
- `src/runtime/bridge/context/evaluator/SceneEvaluator.ts` 目前只覆盖变量声明。
- `src/runtime/bridge/context/evaluator/Evaluator.ts` 基本只会处理字面量。

这意味着当前最核心的问题不是“继续设计更多语法”，而是“把已经设计好的语义真正跑起来”。

### 2. 产品叙事已经写出“静态生成 TypeScript”，但 compiler 还不存在

`docs/zh/0. 基本.md` 已明确把 NarraLang 定义为“DSL 转 TypeScript 的静态脚本生成”，但仓库里目前没有独立 compiler 实现。

这导致一个直接后果：

- NarraLang 目前更像“有 parser 的语言草案”，而不是“有完整产物链的语言工具”。

### 3. parser 已经够强，但还没有真正成为生态底座

从 `src/core/**` 和 `test/parser/**` 看，lexer/parser 是当前最成熟的部分。

但它还没有向下游输送三个稳定产物：

- 稳定 AST 契约
- 稳定诊断模型
- 稳定工具消费接口

缺这三者，后续的 `runtime`、`nlc`、`NarraLint` 都会反复返工。

### 4. 很多“内建能力”还停留在文档层

目前文档里已经写出的这些内容，大多还没有统一执行或生成语义：

- `local set`
- 内建对象，如 `Math`、`Date`、`Random`
- 内建语法糖，如 `add`、`remove`
- `Promise` 与 `await`
- 场景交互语法糖

其中有些是“未来设计”，但有些已经足够具体，应该尽快变成真实能力。

## 方向总览

下面是我认为最值得保留的 8 个方向，已经按当前阶段的优先级做了排序。

| 排名 | 方向 | 类型 | 价值判断 | 复杂度 |
|------|------|------|------|------|
| 1 | `nlc` 最小闭环 | 缺失核心功能 | 必须补齐 | 中 |
| 2 | runtime 语句执行闭环 | 缺失核心功能 | 必须补齐 | 高 |
| 3 | 表达式求值与常量折叠 | 缺失核心功能 | 必须补齐 | 中 |
| 4 | 统一诊断模型与 NarraLint 基座 | 仓库内工具 | 高价值 | 中 |
| 5 | 多文件模块与项目级编译 | 产品能力 | 高价值 | 高 |
| 6 | 富文本对白资产管线 | 竞争力功能 | 高价值 | 中 |
| 7 | 内建语法糖与宿主绑定契约 | 竞争力功能 | 中高价值 | 中 |
| 8 | 运行时状态序列化与恢复 | runtime 能力 | 中价值 | 中 |

## 详细方向评估

## 1. `nlc` 最小闭环

### 方向定义

在本仓库中落地真正的 `NarraLang Compiler`：

- 输入 `.nls`
- 调用 `core` 生成 AST
- 输出静态 TypeScript 文本
- 提供最小 CLI

### 这是在补什么缺口

- 补上文档中已经明确承诺、但工程上尚未存在的主能力。
- 让 NarraLang 从“能 parse”变成“能产出可用脚本”。

### 为什么它最重要

- 这是产品定义的一部分，不是额外增强。
- 它能直接验证语言设计是否真正可落地。
- 它会迫使你把 AST、语义映射、错误输出都做实。

### 为什么它有竞争力

相比只能在特定引擎里解释执行的叙事 DSL，能够生成静态 TypeScript 的 NarraLang 会更适合：

- 纳入正常前端工程构建链
- 接入 CI
- 结合类型系统
- 进行产物审查与版本控制

这会让 NarraLang 更像一个“工程化语言工具链”，而不只是“脚本方言”。

### 仓库内可交付内容

- `@narraleaf/nlc` 包
- 编译 API
- 最小 CLI
- 生成结果测试
- 错误输出格式

### 推荐边界

第一版先不要做太多：

- 先支持单文件编译
- 先覆盖文档中的核心语法子集
- 先输出可读 TS 文本
- 暂不追求复杂优化

### 复杂度

中。难点不在“写个 CLI”，而在“为语义落地建立第一条真实主线”。

## 2. runtime 语句执行闭环

### 方向定义

把 parser 已支持的语句，系统性映射为 NarraLeaf runtime 可执行的节点树或动作链。

重点包括：

- 场景声明
- 对话节点
- 变量声明与赋值
- `local` 作用域
- 控制流
- 语法糖调用

### 这是在补什么缺口

当前 runtime 最大的问题是“入口存在，执行闭环不存在”。

具体表现：

- `src/runtime/index.runtime.ts` 为空
- `Bridge.constructNodeTree()` 为空
- `SceneEvaluator` 只实现了极少数节点类型

### 为什么它重要

如果 runtime 不补全，NarraLang 即使完成 workspace 化，也仍然不能真正驱动 NarraLeaf 运行。

### 为什么它有竞争力

NarraLang 的差异化不只是“能写 DSL”，而是“能和 NarraLeaf 运行时紧密耦合又保持语义清晰”。

一旦你把 AST -> action/node 映射做完整，就会形成一个非常强的垂直一体化优势：

- 脚本语言
- 引擎运行时
- 语义作用域
- 叙事节点执行

这些都在同一仓库内可控。

### 仓库内可交付内容

- `@narraleaf/runtime` 包的正式入口
- `SceneEvaluator` 扩展
- `Bridge` 落地
- 场景与上下文动作完整映射
- runtime 级测试

### 推荐边界

第一阶段不要追求插件化架构，先把最常用 AST 节点真正跑通。

### 复杂度

高。因为它涉及语义、作用域、错误传播和引擎适配。

## 3. 表达式求值与常量折叠

### 方向定义

扩展 `Evaluator`，让其真正覆盖 parser 已支持的表达式能力。

至少包括：

- 标识符读取
- 二元/一元表达式
- 成员访问
- 调用表达式
- 数组/对象/元组构造
- 条件表达式

同时在 `nlc` 路线中逐步引入“常量折叠”。

### 这是在补什么缺口

当前 runtime 在表达式上几乎只支持字面量，这会直接阻断：

- 非字面量变量赋值
- 条件判断
- 一部分对白插值
- 函数参数计算
- 大多数实用脚本逻辑

### 为什么它重要

这是 parser 与 runtime/compiler 之间最关键的中间层。

没有它：

- runtime 只能跑最浅层脚本
- nlc 也很难做稳定生成

### 为什么它有竞争力

表达式求值成熟后，NarraLang 才真正具备“既适合作者写，又不失逻辑能力”的平衡点。

### 仓库内可交付内容

- `Evaluator.evaluateExpression()` 完整化
- `DataType` 与表达式映射补齐
- 常量表达式求值
- 表达式级测试矩阵

### 复杂度

中。范围广，但边界清晰，且可以分批完成。

## 4. 统一诊断模型与 NarraLint 基座

### 方向定义

在仓库内建立统一的 `Diagnostic` 模型，让以下内容共享同一套错误/警告结构：

- parser 错误
- compiler 错误
- lint 规则
- 运行时前置校验

然后在此基础上落地 `NarraLint` 的最小版本。

### 这是在补什么缺口

当前已有 `ParserError`，但还没有统一扩展为完整工具链可共享的诊断协议。

### 为什么它重要

一旦进入多包阶段，若错误模型不统一，会导致：

- CLI 输出不一致
- 测试断言困难
- 未来工具难以复用核心能力

### 为什么它有竞争力

真正专业的语言工具链，不只是“能编译”，还包括“出错时讲得清楚”。

对叙事脚本作者尤其如此：

- 可定位
- 可读
- 可批量分析
- 可做项目级质量规则

### 仓库内可交付内容

- 统一 `Diagnostic` 结构
- 错误码与严重级别
- `NarraLint` 最小规则集
- CLI 输出格式如 text/json

### 建议的第一批规则

- 未使用声明
- 危险循环提示
- 不可达分支提示
- 不支持的宿主调用提示
- 文档语义与运行时不一致的用法提示

### 复杂度

中。基础设施价值很高，但前提是 AST 边界先稳定。

## 5. 多文件模块与项目级编译

### 方向定义

从“单文件脚本”升级到“项目级脚本组织”。

重点包括：

- 多 `.nls` 文件编译
- 模块边界
- 导入导出语义
- 编译依赖图
- 批量编译与 watch

### 这是在补什么缺口

当前仓库还没有清晰的项目级编译模型，`Module` 相关 runtime 结构也还处于空壳状态。

### 为什么它重要

没有多文件能力，NarraLang 很难服务真正的中大型叙事项目。

### 为什么它有竞争力

很多 DSL 在单文件 demo 上很好看，但一进入真实项目就失去组织能力。

如果 NarraLang 能较早建立稳定的项目级编译模型，会显著增强它的可用性和专业感。

### 仓库内可交付内容

- 模块语义设计
- 多文件编译器输入模型
- watch 模式
- 编译缓存或依赖追踪
- 项目级测试用例

### 复杂度

高。因为它会影响 compiler、runtime 和未来的 package 边界。

## 6. 富文本对白资产管线

### 方向定义

围绕已经存在的富文本字符串与对白结构，建立更完整的仓库内能力：

- 对白富文本 AST 校验
- 标签合法性检查
- 可抽取文本内容
- 可分离文本与样式语义
- 为未来本地化做结构准备

### 这是在补什么缺口

当前 parser 已经有 `parseRichString` 这样的能力苗头，但它还没有被提升为稳定产品能力。

### 为什么它重要

对白是叙事 DSL 的核心载体，不只是普通字符串。

### 为什么它有竞争力

这是 NarraLang 很有潜力做出差异化的地方。

相比通用脚本语言，NarraLang 可以天然支持：

- 文本样式标签
- 文本内插值
- 叙事对白结构化
- 后续文本抽取和本地化准备

这种“对白本身就是结构化资产”的能力，非常适合成为语言卖点。

### 仓库内可交付内容

- 富文本标签校验器
- 文本抽取 API
- 对白 AST 标准化结构
- 本地化预处理接口

### 复杂度

中。功能价值高，而且非常贴近产品特色。

## 7. 内建语法糖与宿主绑定契约

### 方向定义

把文档中写明的这些能力，从“解析得出来”升级为“有明确执行/生成语义”：

- `add/sub/mul/div`
- `insert/remove`
- `local set`
- 内建对象如 `Math`、`Date`、`Random`
- 宿主对象与保留全局名

### 这是在补什么缺口

目前这些内容大多是：

- 文档定义较清楚
- parser 可以接受
- runtime 与 compiler 还没有统一解释

### 为什么它重要

如果这些“内建能力”长期没有统一契约，后面会出现两个问题：

- runtime 和 nlc 对同一语法理解不同
- 错误信息与用户心智越来越分裂

### 为什么它有竞争力

叙事 DSL 的强项不是语法多，而是语法背后有明确、稳定、贴近领域的语义。

把这些能力契约化之后，NarraLang 会更像一门“专门为叙事工程设计的语言”，而不是“换皮 JavaScript”。

### 仓库内可交付内容

- sugar 注册表或规范化中间表示
- 内建全局契约表
- runtime lowering
- compiler lowering
- lint 规则支持

### 复杂度

中。关键在于先统一契约，不急着一口气实现全部宿主能力。

## 8. 运行时状态序列化与恢复

### 方向定义

补全 runtime 侧的状态持久化与恢复能力，例如：

- 当前场景上下文
- 场景变量
- `local` 作用域状态
- 必要的 module/runtime 元信息

### 这是在补什么缺口

当前 `ContextActionsService.serialize()` 返回 `null`，`deserialize()` 也还没落地。

### 为什么它重要

对于叙事 runtime 来说，状态恢复不是锦上添花，而是很实际的能力：

- 存档
- 重进场景
- 调试回放
- 运行时恢复

### 为什么它有竞争力

如果 NarraLang runtime 本身就具备较完整的状态恢复设计，它会比许多“只能即时解释执行”的脚本层更适合真实项目。

### 仓库内可交付内容

- runtime 状态结构
- 序列化/反序列化协议
- 场景变量恢复测试
- 与 NarraLeaf 生命周期的恢复对齐

### 复杂度

中。不是第一优先级，但很适合在 runtime 第一轮闭环后补上。

## 推荐分层路线

为了避免方向很多但推进发散，我建议把它们按三层理解。

### A. 必须先补齐的能力

- `nlc` 最小闭环
- runtime 语句执行闭环
- 表达式求值与常量折叠

这三项决定 NarraLang 能不能真正“从语言设计走向可用产品”。

### B. 会显著增强专业度的能力

- 统一诊断模型与 NarraLint 基座
- 多文件模块与项目级编译
- 运行时状态序列化与恢复

这三项决定 NarraLang 能不能从 demo 级工具迈向工程级工具链。

### C. 能形成明显产品差异化的能力

- 富文本对白资产管线
- 内建语法糖与宿主绑定契约

这两项决定 NarraLang 会不会只是“另一个脚本语法”，还是“真正面向叙事内容生产的语言”。

## 现在不建议优先做的方向

下面这些方向不是永远不做，而是当前阶段不适合优先。

### 1. 继续快速扩张语法表面积

原因：

- parser 已经够丰富了。
- 现在真正缺的是语义落地，不是更多表层语法。

### 2. 过早做大型插件体系

原因：

- runtime API 还没稳定。
- sugar 与宿主契约还未固定。

### 3. 把 `@narraleaf/share` 做成大而全公共工具箱

原因：

- 会重新制造耦合。
- 会拖慢边界澄清。

### 4. 先做很重的 REPL、Playground、炫技型演示功能

原因：

- 当前核心闭环都还没打通。
- 演示面越大，和真实能力的落差越难维护。

### 5. 过早做深度优化或额外 IR/字节码

原因：

- 目前更需要真实编译链和真实 runtime 闭环。
- 过早抽象会放大返工。

## 最推荐的近期组合

如果从“未来一年最值得投的组合”来选，我会建议按下面顺序推进：

1. `nlc` 最小闭环
2. runtime 语句执行闭环
3. 表达式求值与常量折叠
4. 统一诊断模型与 NarraLint 基座
5. 富文本对白资产管线

这组组合的好处是：

- 先补齐主产品闭环
- 再提升工程专业度
- 最后强化语言特色

## 一句话结论

NarraLang 未来最值得发展的，不是更多“表面语法”，而是三条真正能形成产品闭环的能力线：**静态编译链、运行时执行闭环、结构化叙事资产能力**；只有这三条线建立起来，后续的 lint、模块化、多文件工程和高级工具才会真正有价值。
